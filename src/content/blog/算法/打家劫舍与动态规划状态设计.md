打家劫舍是动态规划里非常经典的一题。

它本身不难，但它特别适合拿来练一个更重要的能力：

- 状态到底该怎么定义
- 为什么有些题适合定义成“到 `i` 为止”
- 为什么有些题更适合定义成“以 `i` 结尾”

很多时候，题目难不在转移方程，而在你一开始把 `dp[i]` 想成了什么。

这篇文章就围绕这个点展开。

## 题目本质

打家劫舍的题意很简单：

- 有一排房子
- 每间房有金额 `nums[i]`
- 不能偷相邻的两间
- 求最多能偷多少钱

例如：

```text
nums = [2, 7, 9, 3, 1]
```

最优方案是偷：

```text
2 + 9 + 1 = 12
```

## 第一种想法：到第 i 个位置为止的整体最优

这是这题最常见、也最自然的状态定义。

我们定义：

```text
dp[i] = 考虑前 i 个位置（下标 0 到 i）时，能得到的最大金额
```

注意这里的关键词是：

**到 `i` 为止的整体最优**

它不要求你必须偷第 `i` 家，也不要求你必须不偷第 `i` 家。

它只关心一件事：

**在前 `i` 个位置这个范围里，最好的结果是多少。**

### 为什么这样定义很自然

因为当你走到第 `i` 家时，实际上只有两种决策：

1. 不偷第 `i` 家
2. 偷第 `i` 家

如果不偷第 `i` 家，那么答案就是：

```text
dp[i - 1]
```

如果偷第 `i` 家，那么第 `i - 1` 家不能偷，所以答案就是：

```text
dp[i - 2] + nums[i]
```

于是得到转移：

```text
dp[i] = max(dp[i - 1], dp[i - 2] + nums[i])
```

这就是标准解法。

### Java 写法

```java
class Solution {
    public int rob(int[] nums) {
        int n = nums.length;
        if (n == 1) {
            return nums[0];
        }

        int[] dp = new int[n];
        dp[0] = nums[0];
        dp[1] = Math.max(nums[0], nums[1]);

        for (int i = 2; i < n; i++) {
            dp[i] = Math.max(dp[i - 1], dp[i - 2] + nums[i]);
        }

        return dp[n - 1];
    }
}
```

### 这种定义的优点

- 状态含义直接对应题目要的答案
- 转移只依赖前两个状态
- 最终答案就是 `dp[n - 1]`

这类定义通常是线性 DP 里最顺手的一种。

## 第二种想法：必须偷第 i 个位置时的最大值

很多人第一次想这题时，会自然想到另一种定义：

```text
dp[i] = 在必须偷第 i 家的前提下，能得到的最大金额
```

这个定义其实也没错。

只是它会比第一种更绕。

### 这个定义下怎么转移

如果你必须偷第 `i` 家，那么第 `i - 1` 家一定不能偷。

所以前一个合法位置可能是：

- `i - 2`
- `i - 3`
- `i - 4`
- ...

也就是说：

```text
dp[i] = max(dp[j]) + nums[i]，其中 j <= i - 2
```

直接写出来像这样：

```java
class Solution {
    public int rob(int[] nums) {
        int n = nums.length;
        int[] dp = new int[n];
        int ans = 0;

        for (int i = 0; i < n; i++) {
            dp[i] = nums[i];
            for (int j = 0; j <= i - 2; j++) {
                dp[i] = Math.max(dp[i], dp[j] + nums[i]);
            }
            ans = Math.max(ans, dp[i]);
        }

        return ans;
    }
}
```

这个版本是对的，但它的时间复杂度是：

```text
O(n^2)
```

### 为什么这种定义更绕

因为它描述的是：

**以 `i` 为结尾，并且必须选 `i` 的最优值**

而题目最后要的其实是：

**整个数组范围内的整体最优值**

所以这个定义会带来两个问题：

1. `dp[n - 1]` 不一定是答案  
   因为最优解可能根本不偷最后一家

2. 转移不够局部  
   你不能只看 `i - 1` 和 `i - 2`，而要看更前面一大段状态

### 能不能把这种定义也优化？

可以。

如果你进一步定义：

```text
best[i] = 前 0..i 范围内，dp[j] 的最大值
```

那么就可以把：

```text
dp[i] = max(dp[j]) + nums[i]，其中 j <= i - 2
```

优化成：

```text
dp[i] = best[i - 2] + nums[i]
```

但你会发现，推到最后，本质上你还是在维护“前缀最优”。

也就是说，这条路最终还是会绕回第一种定义。

这也是为什么标准解法更愿意一开始就把状态定义成：

```text
到 i 为止的整体最优
```

## 两种定义到底差在哪

我们把它们并排看一下。

### 定义一：前缀整体最优

```text
dp[i] = 前 0..i 范围内的最优解
```

特点：

- 更像题目最终在问什么
- 通常答案直接是 `dp[n - 1]`
- 很适合“选或不选当前元素”这类题

### 定义二：以 i 结尾 / 必须选 i

```text
dp[i] = 以 i 为结尾的最优解
```

或者：

```text
dp[i] = 必须选第 i 个元素时的最优解
```

特点：

- 更适合描述“最后一步一定落在 i”
- 很多时候最后答案要再对所有 `dp[i]` 取最大值
- 常出现在子数组、子序列、路径结尾类问题里

一句话概括：

**前缀整体最优，描述的是一个范围的最好结果；以 `i` 结尾的最优，描述的是一种“必须落在这里”的结构。**

## 什么时候更适合“到 i 为止”

如果题目在问：

- 前 `i` 个位置整体最优是多少
- 当前元素只有“选 / 不选”两种决策
- 最终答案就是整个前缀的最优值

那么通常更适合定义成：

```text
dp[i] = 到 i 为止的整体最优
```

### 典型例子 1：打家劫舍

```text
dp[i] = max(dp[i - 1], dp[i - 2] + nums[i])
```

### 典型例子 2：爬楼梯

定义：

```text
dp[i] = 到第 i 阶的方法数
```

转移：

```text
dp[i] = dp[i - 1] + dp[i - 2]
```

这里 `dp[i]` 也是“到 i 为止”的意思，而不是“必须从哪一步跳过来”。

### 典型例子 3：最小花费爬楼梯

定义：

```text
dp[i] = 到达第 i 阶的最小花费
```

转移：

```text
dp[i] = min(dp[i - 1] + cost[i - 1], dp[i - 2] + cost[i - 2])
```

它也是一个非常标准的前缀整体最优。

## 什么时候更适合“以 i 结尾”

如果题目在问：

- 以当前位置为结尾的最长 / 最大结构
- 当前状态天然依赖“前面某个位置接到这里”
- 最终答案不一定落在最后一个位置

那么通常更适合定义成：

```text
dp[i] = 以 i 结尾的最优值
```

### 典型例子 1：最长递增子序列

定义：

```text
dp[i] = 以 nums[i] 结尾的最长递增子序列长度
```

转移：

```text
dp[i] = max(dp[j] + 1)，其中 j < i 且 nums[j] < nums[i]
```

最终答案不是 `dp[n - 1]`，而是：

```text
max(dp[i])
```

这就是典型的“以 `i` 结尾”型 DP。

### 典型例子 2：最大子数组和

定义：

```text
dp[i] = 以 nums[i] 结尾的最大子数组和
```

转移：

```text
dp[i] = max(nums[i], dp[i - 1] + nums[i])
```

这题如果定义成“前 `i` 个元素整体最大子数组和”，反而不太自然；
定义成“以 `i` 结尾”就非常顺。

最终答案同样是：

```text
max(dp[i])
```

### 典型例子 3：最长连续递增序列

定义：

```text
dp[i] = 以 nums[i] 结尾的最长连续递增长度
```

如果：

```text
nums[i] > nums[i - 1]
```

那么：

```text
dp[i] = dp[i - 1] + 1
```

否则：

```text
dp[i] = 1
```

这也很明显是“以 `i` 结尾”的思路。

## 怎么快速判断该用哪一种定义

我自己做题时，会先问两个问题。

### 问题一：题目最后要的答案，是不是一个前缀整体最优？

如果是，优先想：

```text
dp[i] = 到 i 为止的最优解
```

比如：

- 打家劫舍
- 爬楼梯
- 最小花费爬楼梯
- 股票买卖中的某些状态机题

### 问题二：题目是否强调“以当前位置结束”？

如果是，优先想：

```text
dp[i] = 以 i 结尾的最优解
```

比如：

- 最长递增子序列
- 最大子数组和
- 最长连续递增序列

## 一个很实用的经验

如果你发现自己定义了一个状态以后，出现下面这些现象：

- 最终答案还要额外再绕一层才能拿到
- 转移要从一大堆前置状态里扫最大值
- 你总觉得题意和状态之间隔了一层

那通常说明：

**这个状态定义虽然不一定错，但不够顺。**

打家劫舍就是一个非常典型的例子。

“必须偷第 `i` 家”当然能做，但它没有“到 `i` 为止的整体最优”那么自然。

## 总结

打家劫舍最值得练的，不是那条公式，而是状态设计这件事。

这题里更好的定义是：

```text
dp[i] = 到第 i 个位置为止的整体最优
```

因为它直接对应题目目标，而且转移很局部：

```text
dp[i] = max(dp[i - 1], dp[i - 2] + nums[i])
```

而“必须偷第 `i` 家”的定义也能做，但它更像：

```text
dp[i] = 以 i 为结尾的某种方案
```

这种定义更适合 LIS、最大子数组和这类“答案不一定落在最后，但状态必须落在当前位置”的题。

最后你可以把两类状态定义记成一句话：

- **问前缀整体最优，就优先想“到 i 为止”**
- **问以当前位置结束的结构，就优先想“以 i 结尾”**

动态规划很多时候不是公式难，而是状态定义一旦选对，后面的转移会自己长出来。
