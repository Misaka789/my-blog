# RAG 架构工程实践：从基础原理到多路召回与复杂场景剖析

## 零、RAG 是什么？为什么需要 RAG？

> 面试第一问几乎必问："RAG 是什么？它解决了 LLM 的什么问题？"

**RAG（Retrieval-Augmented Generation，检索增强生成）** 是一种将 **外部知识检索** 与 **LLM 生成** 结合的架构范式。核心流程：

```
用户提问 → 检索相关文档片段（Retrieve） → 将检索结果注入 Prompt 上下文 → LLM 基于上下文生成回答（Generate）
```

**它解决了 LLM 的三大核心痛点：**

1. **知识截止（Knowledge Cutoff）**：LLM 训练数据有截止日期，无法感知最新信息。RAG 可以实时检索最新文档。
2. **幻觉问题（Hallucination）**LLM 在缺乏知识时会编造答案。RAG 提供了真实的上下文依据，显著降低幻觉率。
3. **私有知识盲区**：LLM 无法访问企业内部数据。RAG 可对接私有知识库，实现领域化应答。

**RAG vs Fine-tuning 的选型（高频面试题）：**

| 维度 | RAG | Fine-tuning |
|------|-----|-------------|
| 知识更新 | 实时，改文档即生效 | 需重新训练，成本高 |
| 成本 | 低，无需 GPU 训练 | 高，需标注数据 + 训练资源 |
| 可解释性 | 强，可追溯到引用来源 | 弱，知识内化在参数中 |
| 适用场景 | 知识密集型 QA、文档检索 | 风格迁移、特定任务优化 |
| 幻觉控制 | 更好，有外部证据兜底 | 较弱，仍可能编造 |

**结论**：大多数企业场景优先选 RAG；Fine-tuning 适合对输出格式和风格有强约束的场景。二者也可组合使用。

## 一、Chunking 策略：文档切块的核心设计

> 面试高频题："你的 Chunk 策略是怎么设计的？Chunk 大小怎么定？"

Chunking 是 RAG 管线中最影响最终效果的环节之一。切块质量直接决定检索召回率和生成质量。

### 1.1 常见切块方法

- **固定长度切块（Fixed-size Chunking）**：按 Token 数或字符数切割，设置 overlap（重叠区）防止上下文断裂。简单高效，但会打断语义。
- **递归字符切割（Recursive Character Splitting）**：LangChain 默认方案，按 `\n\n` → `\n` → `. ` → ` ` 的优先级递归分割，尽量保留段落完整性。
- **语义切块（Semantic Chunking）**：使用 Embedding 模型计算相邻句子的相似度，当相似度骤降时切分。效果最好，但计算成本最高。
- **文档结构感知切块**：利用 Markdown 标题层级、PDF 章节目录等结构信息切块。适合结构化文档。

### 1.2 Chunk 大小的权衡

- **Chunk 太小**（< 128 tokens）：丢失上下文，检索到碎片信息，LLM 无法理解完整含义。
- **Chunk 太大**（> 1024 tokens）：引入噪声，召回精度下降；且占用更多 Prompt Token 预算。
- **业界经验值**：通常 256-512 tokens，overlap 设 10%-20%。具体需根据文档类型和 Embedding 模型的最大输入长度调优。

### 1.3 面试加分点：Parent-Child Chunk 策略

检索时用小 Chunk（细粒度）提高精度，返回给 LLM 时用大 Chunk（父级）保留上下文。这是 LlamaIndex 中 `SentenceWindowNodeParser` 的核心思路。

## 二、Embedding 模型选择与向量检索

> 面试题："你用什么 Embedding 模型？为什么选它？"

### 2.1 模型选型

- **开源首选**：BGE 系列（BAAI）、GTE 系列（阿里）、M3E（中文优化）。
- **闭源 API**：OpenAI `text-embedding-3-small/large`，性能好但有数据外传风险。
- **选型依据**：参考 MTEB（Massive Text Embedding Benchmark）排行榜，关注 Retrieval 维度得分。

### 2.2 向量数据库选型

| 数据库 | 特点 |
|--------|------|
| Milvus | 分布式，支持万亿级向量，Pre-filtering 成熟 |
| Qdrant | Rust 编写，性能好，支持标量+向量混合过滤 |
| Weaviate | 内置 BM25+向量混合搜索 |
| Chroma | 轻量级，适合原型验证 |
| pgvector | PostgreSQL 插件，适合已有 PG 基础设施的团队 |

## 三、RAG 多路召回的核心通道与执行机制

在 RAG 架构中，单一的向量检索难以应对专有名词、逻辑推理和实时信息的挑战。工业界通常采用 **多路召回（Multi-channel Retrieval）** 架构。

### 3.1 核心检索通道

1. **稀疏检索（BM25/关键词检索）**：基于 Elasticsearch 的 BM25 算法，对精确术语和专有名词召回率极高。这是**混合检索的基线通道**，面试中不可忽略。

2. **稠密向量检索（Dense Retrieval）**：通过 Embedding 模型将 Query 和 Chunk 映射到同一向量空间，计算余弦相似度。擅长语义理解，但对精确关键词匹配较弱。

3. **知识图谱检索（Knowledge Graph Retrieval）**：依托 Neo4j 等图数据库，专解实体关系和多跳推理问题，弥补向量检索在强逻辑关联场景下的低召回率。

4. **外部 API 与互联网搜索**：通过 Google Search API 或企业内部系统 OpenAPI，获取实时性极强的数据或用户私有动态数据。

5. **语义缓存检索（Semantic Cache Retrieval）**：利用 Redis + 向量相似度搜索（如 Redis Stack 的 VSS 模块）作为最前置通道，拦截高频重复问题，降低底层数据库并发压力与 LLM 开销。

### 3.2 意图识别与路由（Query Routing）

检索前置阶段，通过分类模型或 LLM 提示词判断用户意图，精准路由至对应检索通道，避免无效的全量查询。例如：
- "张三是谁的领导？" → 路由至知识图谱
- "今天黄金价格" → 路由至实时 API
- "公司报销流程" → 路由至向量 + BM25 混合检索

### 3.3 并行召回与重排（Reranking）

并发执行多路查询降低延迟，随后分两阶段融合：
- **粗排**：利用 **RRF（Reciprocal Rank Fusion，倒数排序融合）** 算法融合各路排名。
- **精排**：通过 **Cross-Encoder** 模型（如 BGE-Reranker、Cohere Rerank）对候选集逐对打分截断。

Cross-Encoder 与 Bi-Encoder 的区别（面试常问）：
- **Bi-Encoder**：Query 和 Document 独立编码，速度快，适合召回阶段。
- **Cross-Encoder**：Query 和 Document 拼接后联合编码，精度高但速度慢，适合重排阶段。

## 四、Chunk 元数据的存储设计与权衡

关于 Chunk 及其元数据（如 chunk_id、原文、时间戳、作者等）的存储，需根据数据量和检索复杂度进行架构选择。

### 方案一：全量存入向量数据库（轻量级方案）

将 Embedding 向量、原文及所有元数据存入 Milvus 或 Qdrant。架构极简，但对复杂标量过滤支持弱，长文本存储成本极高。**适合 PoC 或小规模场景。**

### 方案二：严格的冷热分离（大规模基础方案）

向量库仅存向量数组和 chunk_id；MySQL 或 ES 存储原文与业务元数据。先向量召回 ID，再去关系型数据库回表。此方案容易在附带过滤条件时引发 **漏斗问题**。

### 方案三：混合存储与标量前置过滤（业界主流方案）

将用于检索过滤的 **轻量级元数据**（如租户 ID、时间、分类）与向量一同存入向量数据库，以支持 Pre-filtering。不参与检索过滤的 **重量级原文和复杂关系数据** 存放于 MySQL 或对象存储，通过 chunk_id 关联读取。

## 五、架构一致性与极端场景应对指南

在"MySQL 存储原文 + 向量数据库存储特征"的双写架构中，需针对分布式系统的容错能力和数据一致性做底层兜底。

### 5.1 MySQL 与向量数据库的数据一致性保障

- **增量同步机制**：业务系统仅向 MySQL 写入数据。通过 Canal 或 Debezium 监听 MySQL Binlog，将数据变更投递至 Kafka 等消息队列。
- **异步向量化**：消费端订阅消息队列，调用 Embedding 模型生成向量后写入向量数据库，实现以 MySQL 为主库的 **最终一致性**，且不阻塞业务主干流程。
- **异常兜底与对账**：消费端写入失败时利用 **死信队列（DLQ）** 触发告警。同时部署每日定时巡检任务，比对 MySQL 与向量库的 chunk_id，自动补偿遗漏或清理脏数据。

### 5.2 矛盾 Chunk 信息的拦截与处理

- **元数据升维干预**：在重排阶段读取 Chunk 的 `version` 或 `update_time` 标量，通过业务规则强制剔除旧版本数据，仅保留最新描述。
- **来源权重覆盖**：为文档打上 `authority_level` 标签。当高权重来源（如官方文档）与低权重来源（如社区讨论）发生冲突时，程序层面直接丢弃低权重 Chunk。
- **Prompt 强制约束**：在 System Prompt 中植入防幻觉指令，要求 LLM 在面临无法程序分辨的冲突信息时，必须向用户明确陈述矛盾点，严禁自行编造。

### 5.3 标量过滤的漏斗问题避坑

**严禁使用 Post-filtering（后置过滤）**。若先查出 Top 100 向量再回 MySQL 过滤年份或分类，极易导致召回结果清零。

**必须采用 Pre-filtering（前置过滤）**，在发起向量检索时直接将标量条件传入向量数据库，在其底层的 HNSW 索引图与倒排索引中联合遍历，确保输出的 Top K 绝对满足业务条件。

### 5.4 文档更新导致的 Chunk 边界断裂问题

放弃句子级别的细粒度更新策略。

采用 **文档级覆盖更新**：当原文档发生局部修改导致字数或段落变化时，异步任务将原 `document_id` 关联的旧 `chunk_id` 全部软删除。随后对新文档重新执行切块与 Embedding 流程，生成全新 Chunk，确保语境的绝对完整。

### 5.5 异构检索分数的科学合并

由于 ES 的 BM25 分数（无上限）与向量库的余弦相似度（0-1 浮点数）无法直接比较，需抛弃绝对分数计算。

引入 **RRF（Reciprocal Rank Fusion）倒数排序融合算法**，仅基于各路召回的"排名位置"计分：

```
Score = Σ 1 / (k + rank_i)
```

其中 `k` 通常取 60（经验值），`rank_i` 为该文档在第 i 路召回中的排名。分别计算排序得分后相加，抹平异构系统的算分鸿沟。

## 六、Advanced RAG 技术（面试加分项）

### 6.1 Query 改写与扩展

- **HyDE（Hypothetical Document Embedding）**：让 LLM 先生成一个假设性的"理想答案"，用这个答案做 Embedding 去检索，比直接用短 Query 检索效果更好。
- **Multi-Query**：将用户问题改写为多个不同角度的子问题，分别检索后合并去重。
- **Step-back Prompting**：将具体问题抽象为更高层次的概念性问题，扩大检索范围。

### 6.2 Self-RAG（自省式 RAG）

LLM 在生成过程中自主判断：
1. 是否需要检索？（避免对简单问题无脑检索）
2. 检索到的内容是否相关？（自动过滤噪声）
3. 生成的答案是否被检索内容支持？（自我事实核查）

### 6.3 评估体系（面试必问）

> "你怎么评估 RAG 系统效果？"

使用 **RAGAS** 框架的核心指标：

| 指标 | 含义 | 衡量对象 |
|------|------|----------|
| Faithfulness（忠实度） | 答案是否基于检索到的上下文 | 生成质量 |
| Answer Relevancy（答案相关性） | 答案是否回答了问题 | 生成质量 |
| Context Precision（上下文精度） | 检索到的内容中有多少是相关的 | 检索质量 |
| Context Recall（上下文召回率） | 相关内容被检索到了多少 | 检索质量 |

**端到端评估流程**：构建标注数据集（问题 + 标准答案 + 相关文档），自动化跑评估指标，持续迭代。
