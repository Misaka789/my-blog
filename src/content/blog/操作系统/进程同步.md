## 为什么需要进程同步

当多个进程/线程并发访问共享资源时，如果不加控制，执行结果可能取决于进程的执行顺序，这就是**竞态条件（Race Condition）**。

```c
// 线程 A 和线程 B 同时执行 count++
// count++ 实际上是三步操作：
// 1. 读 count 到寄存器
// 2. 寄存器 +1
// 3. 写回 count
// 交叉执行可能导致结果错误
```

进程同步就是用来协调多个进程的执行顺序，确保共享资源被正确访问。

## 临界区问题

### 基本概念

- **临界资源：** 同一时刻只允许一个进程使用的资源
- **临界区：** 进程中访问临界资源的代码段

### 临界区的访问原则

1. **互斥访问：** 同一时刻只能有一个进程在临界区内
2. **空闲让进：** 临界区空闲时，应允许请求进入的进程立即进入
3. **有限等待：** 请求进入临界区的进程应在有限时间内获得访问权
4. **让权等待：** 不能进入临界区的进程应释放 CPU（非必须，但推荐）

### 软件实现方法

**Peterson 算法（两进程互斥）：**

```c
// 进程 Pi 的代码
flag[i] = true;       // 表示 Pi 想进入临界区
turn = j;             // 让对方先
while (flag[j] && turn == j) {
    // 等待
}
// 临界区
flag[i] = false;
```

满足互斥、空闲让进、有限等待，但不满足让权等待（忙等）。

### 硬件实现方法

**关中断：** 进入临界区前关中断，退出后开中断。简单但不适合多核、用户进程。

**TSL（Test and Set Lock）指令：**

```c
boolean test_and_set(boolean *lock) {
    boolean old = *lock;
    *lock = true;
    return old;
}

while (test_and_set(&lock)) { /* 忙等 */ }
// 临界区
lock = false;
```

**CAS（Compare and Swap）指令：**

```c
boolean compare_and_swap(int *value, int expected, int new_value) {
    int temp = *value;
    if (temp == expected)
        *value = new_value;
    return temp;
}
```

## 信号量（Semaphore）

信号量是 Dijkstra 提出的经典同步机制，由一个整数变量和两个原子操作组成。

### 定义

```c
typedef struct {
    int value;          // 信号量的值
    struct process *L;  // 等待队列
} semaphore;
```

### P 操作（wait / down）

```c
void P(semaphore *S) {
    S->value--;
    if (S->value < 0) {
        // 将当前进程加入等待队列
        block(S->L);
    }
}
```

### V 操作（signal / up）

```c
void V(semaphore *S) {
    S->value++;
    if (S->value <= 0) {
        // 从等待队列唤醒一个进程
        wakeup(S->L);
    }
}
```

### 信号量的含义

- `S->value > 0`：还有 S->value 个资源可用
- `S->value = 0`：没有资源可用，也没有进程等待
- `S->value < 0`：|S->value| 个进程正在等待

### 信号量的分类

**互斥信号量：** 初值为 1，用于实现互斥访问

```c
semaphore mutex = 1;

// 进程 Pi
P(&mutex);
// 临界区
V(&mutex);
```

**同步信号量：** 初值为 0，用于实现进程间的先后顺序

```c
semaphore sync = 0;

// 进程 A（先执行的操作）
执行操作 X;
V(&sync);

// 进程 B（后执行的操作）
P(&sync);
执行操作 Y;  // 保证 Y 在 X 之后执行
```

**计数信号量：** 初值为 n，用于控制对 n 个同类资源的访问

## 经典同步问题

### 1. 生产者-消费者问题

**问题描述：** 生产者向有界缓冲区放入产品，消费者从缓冲区取出产品。缓冲区满时生产者等待，缓冲区空时消费者等待。

```c
semaphore mutex = 1;    // 互斥访问缓冲区
semaphore empty = N;    // 空缓冲区数量
semaphore full = 0;     // 满缓冲区数量

// 生产者
void producer() {
    while (true) {
        生产一个产品;
        P(&empty);       // 等待空缓冲区
        P(&mutex);       // 进入临界区
        放入缓冲区;
        V(&mutex);       // 离开临界区
        V(&full);        // 增加满缓冲区数
    }
}

// 消费者
void consumer() {
    while (true) {
        P(&full);        // 等待满缓冲区
        P(&mutex);       // 进入临界区
        从缓冲区取出;
        V(&mutex);       // 离开临界区
        V(&empty);       // 增加空缓冲区数
        消费产品;
    }
}
```

**注意：** P(empty/full) 必须在 P(mutex) 之前，否则可能死锁！

### 2. 读者-写者问题

**问题描述：** 多个读者可以同时读，但写者必须独占访问。

**读者优先方案：**

```c
semaphore rw_mutex = 1;     // 读写互斥
semaphore count_mutex = 1;  // 保护 read_count
int read_count = 0;         // 当前读者数量

// 读者
void reader() {
    P(&count_mutex);
    read_count++;
    if (read_count == 1)     // 第一个读者需要锁定
        P(&rw_mutex);
    V(&count_mutex);

    读数据;

    P(&count_mutex);
    read_count--;
    if (read_count == 0)     // 最后一个读者释放锁
        V(&rw_mutex);
    V(&count_mutex);
}

// 写者
void writer() {
    P(&rw_mutex);
    写数据;
    V(&rw_mutex);
}
```

**问题：** 如果读者源源不断，写者可能饿死。

**写者优先方案：** 增加一个信号量，一旦有写者等待，后续读者就需要排队。

### 3. 哲学家进餐问题

**问题描述：** 5 个哲学家围坐圆桌，每两人之间有一根筷子。哲学家交替思考和吃饭，吃饭需要同时拿起左右两根筷子。

**可能死锁：** 如果所有哲学家同时拿起左边的筷子，所有人都在等右边的筷子 → 死锁。

**解决方案 1 —— 限制人数：**

```c
semaphore chopstick[5] = {1, 1, 1, 1, 1};
semaphore room = 4;  // 最多 4 人同时进餐

void philosopher(int i) {
    while (true) {
        思考;
        P(&room);
        P(&chopstick[i]);
        P(&chopstick[(i+1) % 5]);
        吃饭;
        V(&chopstick[(i+1) % 5]);
        V(&chopstick[i]);
        V(&room);
    }
}
```

**解决方案 2 —— 奇偶编号：**

- 奇数号哲学家先拿左边再拿右边
- 偶数号哲学家先拿右边再拿左边
- 打破循环等待条件

**解决方案 3 —— 同时拿两根筷子（原子操作）：**

使用互斥锁保证要么同时拿到两根筷子，要么都不拿。

### 4. 吸烟者问题

**问题描述：** 三个吸烟者各有一种材料（烟草、纸、胶水），供应者随机提供两种材料，拥有第三种材料的吸烟者卷烟抽。

这是一个典型的**多资源同步**问题，用信号量可以很好地解决。

## 管程（Monitor）

### 为什么需要管程

信号量强大但容易出错：P/V 操作的顺序和配对一旦写错就可能死锁或互斥失败。管程提供了更高级的同步抽象。

### 管程的定义

管程是一种封装了共享变量和操作的抽象数据类型，包含：

1. **共享数据：** 局部于管程的变量
2. **操作过程：** 一组操作共享数据的函数
3. **初始化代码**
4. **条件变量（Condition Variable）**

### 关键特性

- **互斥性：** 同一时刻只有一个进程可以在管程内执行
- 由编译器或语言运行时保证互斥，程序员不需要手动加锁

### 条件变量

管程中用于进程等待和唤醒的机制：

- `wait(condition)`：当前进程阻塞并释放管程，等待条件成立
- `signal(condition)`：唤醒一个等待该条件的进程

**与信号量的区别：**

| 维度 | 信号量 | 条件变量 |
|------|--------|----------|
| 有值 | 是（整数值） | 否（只有等待队列） |
| wait 操作 | 一定阻塞（S<0时） | 一定阻塞 |
| signal 操作 | 一定唤醒一个（S≤0时） | 有等待者才唤醒 |
| 使用复杂度 | 较高（需小心顺序） | 较低（管程保证互斥） |

### Java 中的管程

Java 的 `synchronized` 就是管程的一种实现：

```java
class BoundedBuffer {
    private final Object[] buffer;
    private int count = 0, in = 0, out = 0;

    public synchronized void put(Object item) throws InterruptedException {
        while (count == buffer.length)
            wait();           // 条件等待
        buffer[in] = item;
        in = (in + 1) % buffer.length;
        count++;
        notifyAll();          // 唤醒等待者
    }

    public synchronized Object get() throws InterruptedException {
        while (count == 0)
            wait();
        Object item = buffer[out];
        out = (out + 1) % buffer.length;
        count--;
        notifyAll();
        return item;
    }
}
```

`ReentrantLock` + `Condition` 也是管程的实现，提供了更灵活的条件变量：

```java
ReentrantLock lock = new ReentrantLock();
Condition notFull = lock.newCondition();
Condition notEmpty = lock.newCondition();

// 生产者
lock.lock();
try {
    while (count == N)
        notFull.await();
    // 放入产品
    notEmpty.signal();
} finally {
    lock.unlock();
}
```

## 同步机制总结

| 机制 | 抽象级别 | 优点 | 缺点 |
|------|---------|------|------|
| 硬件指令（TSL/CAS） | 低 | 简单高效 | 忙等待 |
| 信号量 | 中 | 灵活，可实现互斥和同步 | 容易出错 |
| 管程 | 高 | 安全，编译器保证互斥 | 依赖语言支持 |

## 常见面试问题

**Q：什么是临界区？访问临界区应满足什么条件？**

临界区是进程中访问共享资源的代码段。访问需满足互斥访问、空闲让进、有限等待三个基本原则。

**Q：信号量和互斥锁的区别？**

互斥锁只能由加锁的线程释放（所有权），信号量可以由任何线程执行 V 操作。互斥锁用于互斥，信号量还可以用于同步和资源计数。

**Q：条件变量的 wait 为什么要在 while 循环里？**

因为存在**虚假唤醒（Spurious Wakeup）**：线程可能在条件不成立时被唤醒。用 while 循环可以在被唤醒后重新检查条件。

**Q：如何避免生产者-消费者问题中的死锁？**

确保 P(empty/full) 在 P(mutex) 之前执行。如果先获取互斥锁再等待资源，当缓冲区满/空时，持有互斥锁的进程会阻塞，其他进程也无法进入临界区释放资源，导致死锁。
