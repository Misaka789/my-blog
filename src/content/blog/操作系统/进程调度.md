## 为什么需要进程调度

在多道程序系统中，就绪队列里通常有多个进程等待执行，但 CPU 同一时刻只能运行一个进程（单核情况）。进程调度器负责决定**下一个获得 CPU 的进程是谁**。

调度发生的时机：

- 进程从运行态转为阻塞态（如发起 I/O 请求）
- 进程从运行态转为就绪态（如时间片用完）
- 进程从阻塞态转为就绪态（如 I/O 完成）
- 进程终止

## 调度的层次

| 层次 | 名称 | 频率 | 作用 |
|------|------|------|------|
| 高级调度 | 作业调度 | 低 | 决定哪些作业从外存调入内存（创建进程） |
| 中级调度 | 内存调度 | 中 | 决定哪些进程在内存与外存之间换入换出（挂起/激活） |
| 低级调度 | 进程调度/CPU调度 | 高 | 决定就绪队列中哪个进程获得 CPU |

我们最常讨论的是**低级调度（CPU 调度）**。

## 调度评价指标

- **CPU 利用率：** CPU 忙碌时间占总时间的比例，越高越好
- **吞吐量：** 单位时间内完成的进程数量，越高越好
- **周转时间：** 从进程提交到完成的总时间（等待时间 + 执行时间）
- **等待时间：** 进程在就绪队列中等待的总时间
- **响应时间：** 从提交请求到首次产生响应的时间，交互式系统最关注

## 抢占式 vs 非抢占式

**非抢占式（Non-preemptive）：**

- 一旦进程获得 CPU，就一直运行到完成或主动放弃（如 I/O 阻塞）
- 简单，但可能导致短进程长时间等待

**抢占式（Preemptive）：**

- 调度器可以强制剥夺当前进程的 CPU（如时间片用完、更高优先级进程到达）
- 响应更及时，但上下文切换开销更大
- 现代操作系统基本都是抢占式

## 常见调度算法

### 1. 先来先服务（FCFS - First Come First Served）

**规则：** 按照进程到达就绪队列的顺序分配 CPU。

**类型：** 非抢占式

**优点：** 简单，公平

**缺点：** 对短进程不友好，可能产生**护航效应（Convoy Effect）**——一个长进程霸占 CPU，导致后面的短进程全部等待。

**示例：**

| 进程 | 到达时间 | 执行时间 | 完成时间 | 周转时间 | 等待时间 |
|------|----------|----------|----------|----------|----------|
| P1   | 0        | 24       | 24       | 24       | 0        |
| P2   | 1        | 3        | 27       | 26       | 23       |
| P3   | 2        | 3        | 30       | 28       | 25       |

平均等待时间 = (0 + 23 + 25) / 3 = 16

### 2. 短作业优先（SJF - Shortest Job First）

**规则：** 优先选择预计执行时间最短的进程。

**类型：** 非抢占式（也有抢占式变体 SRTF）

**优点：** 平均等待时间最小（可证明最优）

**缺点：**

- 难以准确预测进程的执行时间
- 可能导致长进程**饥饿（Starvation）**

### 3. 最短剩余时间优先（SRTF - Shortest Remaining Time First）

**规则：** SJF 的抢占式版本。每当新进程到达时，比较其执行时间与当前进程的剩余时间，更短的优先执行。

**类型：** 抢占式

**优点：** 比 SJF 响应更快

**缺点：** 同样存在饥饿问题，且上下文切换更频繁

### 4. 优先级调度（Priority Scheduling）

**规则：** 每个进程有一个优先级，CPU 分配给优先级最高的进程。

**类型：** 可以是抢占式或非抢占式

**优先级的确定：**

- **静态优先级：** 创建时确定，不再改变
- **动态优先级：** 根据运行情况动态调整（如等待时间越长优先级越高）

**主要问题：** 低优先级进程可能**饥饿**

**解决方案 —— 老化（Aging）：** 随着等待时间增加，逐步提升进程的优先级

### 5. 时间片轮转（RR - Round Robin）

**规则：** 每个进程分配一个固定的时间片（Time Quantum），时间片用完后轮到下一个进程。

**类型：** 抢占式

**时间片大小的影响：**

- **太大：** 退化为 FCFS
- **太小：** 上下文切换过于频繁，开销大
- **经验值：** 通常设为 10-100ms，使得上下文切换开销占比 < 1%

**优点：** 公平，响应时间好，适合交互式系统

**缺点：** 平均周转时间可能不如 SJF

### 6. 多级队列调度（Multilevel Queue）

将就绪队列分为多个独立队列，每个队列有自己的调度算法：

```
高优先级队列（前台/交互进程）→ RR 调度
低优先级队列（后台/批处理进程）→ FCFS 调度
```

- 进程根据属性被永久分配到某个队列
- 队列之间可以采用固定优先级或时间片划分

**缺点：** 进程不能在队列间移动，不够灵活

### 7. 多级反馈队列调度（MLFQ - Multilevel Feedback Queue）

**这是目前公认最好的通用调度算法**，也是大多数现代操作系统（Linux、Windows）采用的思路。

**核规则：**

1. 设置多个优先级从高到低的队列，每个队列的时间片从小到大
2. 新进程进入最高优先级队列
3. 如果进程在当前队列的时间片内完成，正常退出
4. 如果进程时间片用完还没完成，降到下一级队列
5. 如果进程在某一级主动放弃 CPU（如 I/O 等待），回来时留在当前队列或提升

```
队列 0：时间片 8ms（最高优先级）
队列 1：时间片 16ms
队列 2：时间片 32ms
队列 3：FCFS（最低优先级）
```

**优点：**

- 短进程能快速完成（在高优先级队列搞定）
- 长进程不会饥饿（最终能在低优先级队列执行）
- I/O 密集型进程能保持高响应性
- 自适应，不需要提前知道进程执行时间

**缺点：** 参数调优比较复杂（队列数量、时间片大小、升降级策略）

## 调度算法对比

| 算法 | 类型 | 饥饿 | 适用场景 | 复杂度 |
|------|------|------|----------|--------|
| FCFS | 非抢占 | 否 | 批处理 | 低 |
| SJF | 非抢占 | 是 | 已知执行时间 | 中 |
| SRTF | 抢占 | 是 | 已知执行时间 | 中 |
| 优先级 | 都可 | 是 | 有明确优先级 | 中 |
| RR | 抢占 | 否 | 交互式系统 | 低 |
| 多级队列 | 都可 | 可能 | 多类型混合 | 中 |
| MLFQ | 抢占 | 否 | 通用场景 | 高 |

## 实际操作系统的调度

### Linux CFS（Completely Fair Scheduler）

Linux 2.6.23 之后的默认调度器：

- 基于**虚拟运行时间（vruntime）** 概念
- 使用红黑树管理所有就绪进程
- 每次选择 vruntime 最小的进程运行
- 通过 nice 值调整进程权重，nice 越低（优先级越高），vruntime 增长越慢
- 目标：让每个进程获得公平的 CPU 份额

### Windows 调度

- 基于优先级的抢占式调度
- 32 个优先级（0-31），分为实时类（16-31）和普通类（0-15）
- 普通进程有动态优先级提升机制（如窗口获得焦点时提升）

## 常见面试问题

**Q：为什么 MLFQ 被认为是最好的通用调度算法？**

因为它兼顾了短进程的快速响应和长进程的公平执行，不需要预知进程执行时间，能自适应不同类型的工作负载。

**Q：什么是饥饿？怎么解决？**

饥饿是指某个进程长时间得不到 CPU 资源。解决方案：老化机制（随等待时间增加逐步提升优先级）。

**Q：实时调度和普通调度的区别？**

实时调度必须保证任务在截止时间前完成。分为硬实时（必须满足）和软实时（尽量满足）。常见算法：EDF（最早截止时间优先）、RMS（速率单调调度）。
