## 内存管理的目标

操作系统内存管理需要解决的核心问题：

1. **地址转换：** 将程序使用的逻辑地址转换为物理地址
2. **内存分配：** 为进程合理分配内存空间
3. **内存保护：** 防止进程访问其他进程的内存
4. **内存扩充：** 通过虚拟内存技术突破物理内存限制

## 程序的内存布局

一个进程的内存空间（从低地址到高地址）：

```
┌──────────────┐ 高地址
│    内核空间    │
├──────────────┤
│     栈 ↓     │  局部变量、函数调用帧
│              │
│     ↑ 堆     │  动态分配（malloc/new）
├──────────────┤
│   BSS 段     │  未初始化的全局/静态变量
├──────────────┤
│   数据段      │  已初始化的全局/静态变量
├──────────────┤
│   代码段      │  程序指令（只读）
└──────────────┘ 低地址
```

**栈和堆的区别：**

| 维度 | 栈 | 堆 |
|------|------|------|
| 增长方向 | 高地址 → 低地址 | 低地址 → 高地址 |
| 管理方式 | 编译器自动管理 | 程序员手动管理 |
| 分配速度 | 快（移动栈指针） | 慢（需要查找空闲块） |
| 空间大小 | 较小（一般几 MB） | 较大 |
| 碎片问题 | 无 | 有 |

## 连续内存分配

### 固定分区分配

将内存划分为若干固定大小的分区，每个分区装一个进程。

- **优点：** 简单
- **缺点：** 内部碎片严重（分区大小和进程大小不匹配）

### 动态分区分配

根据进程实际需要动态划分分区。

**空闲分区分配算法：**

1. **首次适应（First Fit）：** 从头开始找第一个够大的分区。简单高效，倾向于利用低地址空间。
2. **最佳适应（Best Fit）：** 找大小最接近的分区。碎片最小但查找慢，容易产生大量极小碎片。
3. **最差适应（Worst Fit）：** 找最大的分区。减少极小碎片，但大分区很快被用完。
4. **邻近适应（Next Fit）：** 从上次分配的位置继续查找。分布更均匀。

**外部碎片：** 空闲内存总量够，但不连续，无法分配给进程。

**解决方案 —— 紧凑（Compaction）：** 移动进程使空闲空间合并，但开销大。

## 非连续内存分配

### 分页存储管理

将内存分为固定大小的**页框（Frame）**，将进程分为同样大小的**页面（Page）**。

**核心概念：**

- **页面大小：** 通常 4KB（由硬件决定）
- **页表（Page Table）：** 记录每个页面对应的页框号
- **逻辑地址结构：** 页号 + 页内偏移

**地址转换过程：**

1. 从逻辑地址中提取页号和页内偏移
2. 查页表，找到页号对应的页框号
3. 物理地址 = 页框号 × 页面大小 + 页内偏移

```
逻辑地址：| 页号(20位) | 页内偏移(12位) |
                ↓ 查页表
物理地址：| 页框号     | 页内偏移(12位) |
```

**优点：** 消除了外部碎片

**缺点：** 可能存在页内碎片（最后一个页面未填满）；页表本身占用内存

### 多级页表

一级页表太大怎么办？用多级页表！

**二级页表示例（32位系统，4KB页面）：**

```
逻辑地址：| 一级页号(10位) | 二级页号(10位) | 页内偏移(12位) |
```

- 一级页表有 2^10 = 1024 个条目，指向二级页表
- 二级页表各有 1024 个条目，指向物理页框
- **优势：** 不需要的二级页表可以不创建，节省内存

**64 位系统通常使用四级页表**（如 Linux 的 PGD → PUD → PMD → PTE）。

### TLB（Translation Lookaside Buffer）

TLB 是页表的高速缓存，位于 CPU 内部：

- 存储最近使用的页表项
- 命中 TLB 无需访问内存中的页表
- TLB 命中率通常 > 98%
- **TLB 未命中：** 需要查内存中的页表（多级页表需要多次内存访问）

**带 TLB 的地址转换：**

1. CPU 发出逻辑地址
2. 先查 TLB
3. TLB 命中 → 直接获得物理地址
4. TLB 未命中 → 查页表 → 更新 TLB → 获得物理地址

### 分段存储管理

按程序的逻辑结构划分为不同的段（代码段、数据段、栈段等），每段有独立的地址空间。

**逻辑地址：** 段号 + 段内偏移

**段表：** 记录每个段的基址和长度

**优点：** 符合程序逻辑结构，便于共享和保护

**缺点：** 会产生外部碎片

### 段页式存储管理

结合分段和分页的优点：

1. 先按逻辑分段
2. 每个段内再分页
3. 逻辑地址 = 段号 + 页号 + 页内偏移

**地址转换：** 段表 → 页表 → 物理地址（需要三次内存访问，TLB 可以加速）

## 虚拟内存

### 基本概念

虚拟内存让每个进程以为自己拥有完整的、连续的地址空间，实际上只有部分数据在物理内存中，其余在磁盘上。

**核心思想 —— 局部性原理：**

- **时间局部性：** 最近被访问的数据很可能再次被访问
- **空间局部性：** 被访问地址附近的数据很可能被访问

基于局部性，程序在一段时间内只会用到一小部分页面，不需要全部加载到内存。

### 请求分页

虚拟内存最常见的实现方式：

1. 进程启动时，只加载少量必要页面到内存
2. 访问不在内存中的页面时，触发**缺页中断（Page Fault）**
3. 操作系统将缺失的页面从磁盘换入内存
4. 如果内存已满，需要用**页面置换算法**选择一个页面换出

### 缺页中断处理流程

1. CPU 访问页面，查页表发现有效位为 0（不在内存）
2. 触发缺页中断，陷入内核
3. 操作系统在磁盘上找到该页面
4. 找一个空闲页框（如果没有则执行页面置换）
5. 将页面从磁盘读入该页框
6. 更新页表
7. 重新执行触发缺页的指令

## 页面置换算法

### 1. 最优页面置换（OPT）

**规则：** 淘汰未来最长时间不会被使用的页面。

**特点：** 理论最优，但无法实现（需要预知未来），仅作为对比基准。

### 2. 先进先出（FIFO）

**规则：** 淘汰最先进入内存的页面。

**特点：** 简单，但性能差。可能出现 **Belady 异常**——增加页框数反而导致更多缺页。

### 3. 最近最少使用（LRU - Least Recently Used）

**规则：** 淘汰最近最长时间没有被使用的页面。

**特点：** 性能接近 OPT，但实现开销大（需要记录每个页面的最后访问时间）。

**实现方式：**

- **计数器：** 每次访问记录时间戳，置换时间戳最小的
- **栈：** 访问时将页面移到栈顶，置换栈底的页面
- **近似实现：** 使用引用位（Reference Bit）

### 4. 时钟算法（Clock / Second Chance）

LRU 的近似实现，实际系统中最常用：

- 所有页面组成环形链表，配合一个指针
- 每个页面有一个引用位（初始为 1）
- 需要置换时，检查指针指向的页面：
  - 引用位为 1 → 置为 0，指针前移（给第二次机会）
  - 引用位为 0 → 淘汰该页面

### 5. 改进的时钟算法

在引用位基础上增加**修改位（Dirty Bit）**：

优先级（从最优到最差）：
1. (引用=0, 修改=0)：最佳淘汰对象
2. (引用=0, 修改=1)：需要写回磁盘
3. (引用=1, 修改=0)：最近被访问过
4. (引用=1, 修改=1)：最近访问且被修改

### 页面置换算法对比

| 算法 | 性能 | 实现复杂度 | 特点 |
|------|------|------------|------|
| OPT | 最优 | 不可实现 | 理论基准 |
| FIFO | 差 | 低 | 可能 Belady 异常 |
| LRU | 好 | 高 | 需要硬件支持 |
| Clock | 较好 | 中 | 实际最常用 |

## 内存映射（mmap）

将文件或设备映射到进程的虚拟地址空间，可以像访问内存一样访问文件。

```c
void *addr = mmap(NULL, length, PROT_READ, MAP_PRIVATE, fd, 0);
// 直接通过 addr 读取文件内容
```

**优点：**

- 减少数据拷贝（避免 read/write 的用户态 ↔ 内核态拷贝）
- 多个进程可以共享同一映射区域
- 适合大文件的随机访问

## 内存相关面试问题

**Q：32 位系统最大支持多少内存？**

理论上 2^32 = 4GB。其中用户空间通常 3GB，内核空间 1GB（Linux 默认）。

**Q：malloc 的底层实现？**

- 小内存（< 128KB）：调用 `brk()` 扩展堆
- 大内存（≥ 128KB）：调用 `mmap()` 映射匿名页
- 实际上 glibc 的 malloc 会维护内存池，减少系统调用

**Q：什么是内存泄漏？**

程序分配了内存但没有释放，且丢失了指向该内存的指针，导致这块内存无法被回收。

**Q：什么是页表自映射？**

将页表本身也映射到虚拟地址空间中，这样操作系统可以通过虚拟地址访问和修改页表，无需额外机制。

**Q：什么是写时复制（Copy-on-Write，COW）？**

`fork()` 创建子进程时不立即复制父进程的物理内存，而是让父子进程共享同一段物理内存（只读）。当任一方尝试写入时，才为该页面创建副本。节省内存和 fork 时间。

**Q：什么是内存抖动（Thrashing）？**

频繁的页面换入换出，导致 CPU 大部分时间花在处理缺页中断上，实际有效工作很少。原因通常是进程分配的物理页框太少。
