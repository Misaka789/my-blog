## 为什么需要锁

在多线程/多进程环境中，多个执行流可能同时访问共享资源（如全局变量、文件、数据库记录等），如果不加以控制，就会产生**竞态条件（Race Condition）**，导致数据不一致。锁是解决这个问题的最基本手段。

## 操作系统中的锁

### 自旋锁（Spinlock）

自旋锁是最简单的锁实现。线程在获取锁失败时**不会阻塞挂起**，而是在一个循环中不断检测锁是否被释放（忙等待，busy-waiting）。

**特点：**

- 不涉及线程上下文切换，开销小
- 适合锁持有时间非常短的场景
- 在单核 CPU 上基本无意义（忙等会占满 CPU 时间片）
- 多核场景下效果较好

**伪代码实现：**

```c
while (test_and_set(&lock) == 1) {
    // 忙等待
}
// 临界区
lock = 0; // 释放锁
```

**底层原理：** 依赖硬件提供的原子指令，如 `test_and_set`、`compare_and_swap`（CAS）。这些指令保证了「检测-设置」这个动作是不可分割的。

### 互斥锁（Mutex）

互斥锁在获取失败时会将线程**阻塞挂起**，让出 CPU 给其他线程使用，等到锁释放后由操作系统唤醒。

**特点：**

- 有上下文切换开销
- 适合锁持有时间较长的场景
- 比自旋锁更节约 CPU 资源
- 通常由操作系统内核实现

**自旋锁 vs 互斥锁：**

| 维度 | 自旋锁 | 互斥锁 |
|------|--------|--------|
| 等待方式 | 忙等待 | 阻塞挂起 |
| 上下文切换 | 无 | 有 |
| CPU 占用 | 高（等待期间） | 低 |
| 适用场景 | 短临界区、多核 | 长临界区、任意场景 |

### 读写锁（Read-Write Lock）

读写锁将访问分为**读操作和写操作**两类：

- **读-读不互斥：** 多个线程可以同时读
- **读-写互斥：** 读和写不能同时进行
- **写-写互斥：** 同一时刻只能有一个写线程

**适用场景：** 读多写少的情况，如缓存、配置读取等。

**读写锁的常见策略：**

1. **读优先：** 只要有读锁，新的读请求都能直接获取锁。可能导致写线程饿死。
2. **写优先：** 一旦有写请求在等待，后续读请求必须排队。避免写饥饿但降低读并发。
3. **公平策略：** 按请求到达顺序排队。

### 乐观锁与悲观锁

这两个概念更偏向思想而非具体实现：

**悲观锁：** 假设一定会有冲突，访问前先加锁。互斥锁、读写锁都属于悲观锁。

**乐观锁：** 假设不会有冲突，先直接操作，提交时检查是否有冲突。常见实现方式：

- **版本号机制：** 每次更新带上版本号，提交时检查版本是否匹配
- **CAS（Compare And Swap）：** 比较当前值与期望值，一致则更新

## 死锁

### 死锁产生的四个必要条件

四个条件**缺一不可**，全部满足时才可能产生死锁：

1. **互斥条件：** 一个资源每次只能被一个线程占用
2. **请求与保持条件：** 线程因请求资源而阻塞时，对已获得的资源保持不放（站在进程角度）
3. **不剥夺条件：** 线程已获得的资源，在未使用完之前不能被其他进程强行剥夺，只能自己释放（站在资源角度）
4. **循环等待条件：** 存在线程资源的循环等待关系，即 T1 等待 T2 占用的资源，T2 等待 T3 占用的资源，...，Tn 等待 T1 占用的资源

### 解决死锁的方法

**预防死锁 —— 破坏四个条件之一：**

1. **破坏互斥条件：** 允许多个线程同时访问同一资源（实际中很难做到，某些资源天然互斥）
2. **破坏请求与保持条件：** 要求线程一次性申请所有所需资源，或者在请求新资源前释放已占有资源
3. **破坏不剥夺条件：** 当线程请求资源被拒绝时，主动释放已持有的资源
4. **破坏循环等待条件：** 对所有资源编号，要求线程按编号递增的顺序请求资源

**避免死锁 —— 银行家算法：**

银行家算法是最经典的死锁避免算法。核心思想是在分配资源前，先检查分配后系统是否仍处于**安全状态**（即存在一个安全序列，使所有进程都能完成）。

**检测与恢复死锁：**

- 利用资源分配图检测是否存在环路
- 发现死锁后，终止部分进程或剥夺部分资源来打破死锁

## Java 中的锁

### synchronized 关键字

Java 最基本的加锁方式，基于 **Monitor（监视器锁/管程）** 实现。

```java
// 同步方法
public synchronized void method() {
    // 临界区
}

// 同步代码块
synchronized (lockObject) {
    // 临界区
}
```

**底层原理：**

- 字节码层面：`monitorenter` / `monitorexit` 指令
- 每个对象都有一个 Monitor，包含：
  - 持有锁的线程标识
  - 计数器（支持可重入）
  - 等待队列（EntryList）
  - 条件等待队列（WaitSet）

### synchronized 锁升级

JDK 6 之后，synchronized 做了大量优化，引入了锁升级机制：

```
无锁 → 偏向锁 → 轻量级锁 → 重量级锁
```

**偏向锁（Biased Locking）：**

- 针对只有一个线程访问的场景
- 在对象头 Mark Word 中记录线程 ID
- 后续该线程再次进入时，无需任何同步操作
- 当有其他线程竞争时，撤销偏向锁，升级为轻量级锁

**轻量级锁（Lightweight Lock）：**

- 针对线程交替执行（无实际竞争）的场景
- 通过 CAS 将对象头 Mark Word 替换为指向栈帧中锁记录的指针
- 如果 CAS 成功，获取锁
- 如果 CAS 失败且存在竞争，膨胀为重量级锁

**重量级锁（Heavyweight Lock）：**

- 基于操作系统的 Mutex 实现
- 涉及用户态与内核态切换
- 竞争失败的线程会被阻塞挂起

**锁升级只能单向，不能降级。**

### ReentrantLock

`java.util.concurrent.locks.ReentrantLock` 是显式锁，提供了比 synchronized 更灵活的控制。

```java
ReentrantLock lock = new ReentrantLock();
lock.lock();
try {
    // 临界区
} finally {
    lock.unlock();
}
```

**核心特性：**

- **可重入：** 同一线程可以多次获取同一把锁
- **公平 / 非公平：** 构造时可选择公平模式（先到先得）或非公平模式（允许插队，默认）
- **可中断：** `lockInterruptibly()` 支持在等待锁时被中断
- **超时获取：** `tryLock(timeout)` 支持限时等待
- **条件变量：** 通过 `newCondition()` 创建多个等待队列

**底层实现 —— AQS（AbstractQueuedSynchronizer）：**

- 维护一个 `volatile int state` 表示锁状态
- 维护一个 FIFO 的 CLH 双向队列管理等待线程
- 获取锁：CAS 修改 state
- 释放锁：修改 state 并唤醒队首线程

### ReentrantReadWriteLock

Java 版本的读写锁实现：

```java
ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock();
ReadLock readLock = rwLock.readLock();
WriteLock writeLock = rwLock.writeLock();

// 读操作
readLock.lock();
try {
    // 读取数据
} finally {
    readLock.unlock();
}

// 写操作
writeLock.lock();
try {
    // 修改数据
} finally {
    writeLock.unlock();
}
```

**实现原理：** 利用 AQS 的 state 变量，高 16 位表示读锁计数，低 16 位表示写锁计数。

### CAS 与原子类

**CAS（Compare And Swap）：**

CAS 是一种无锁的乐观并发策略，包含三个操作数：

- 内存位置 V
- 预期原值 A
- 新值 B

只有当 V 的值等于 A 时，才将 V 更新为 B。

**ABA 问题：** 值从 A 变成 B 又变回 A，CAS 会误认为没有变化。解决方案：`AtomicStampedReference`，引入版本号。

**Java 原子类：**

```java
AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet();    // 原子自增
count.compareAndSet(1, 2);  // CAS 操作
```

底层通过 `Unsafe` 类的 `compareAndSwapInt` 等 native 方法实现，最终映射到 CPU 的 `CMPXCHG` 指令。

### volatile 关键字

volatile 本身不是锁，但是 Java 并发中的重要关键字：

- **可见性：** 对 volatile 变量的写入会立即刷新到主存，读取会直接从主存读
- **有序性：** 禁止指令重排序（通过内存屏障实现）
- **不保证原子性：** `volatile int i; i++` 不是线程安全的

**内存屏障（Memory Barrier）：**

- LoadLoad：确保前面的读操作先于后面的读
- StoreStore：确保前面的写操作先于后面的写
- LoadStore：确保前面的读先于后面的写
- StoreLoad：确保前面的写先于后面的读（开销最大，全能屏障）

### synchronized vs ReentrantLock

| 维度 | synchronized | ReentrantLock |
|------|-------------|---------------|
| 实现层级 | JVM 内置 | JDK API 层 |
| 锁释放 | 自动释放 | 需手动 unlock |
| 可中断 | 不支持 | 支持 |
| 超时获取 | 不支持 | 支持 |
| 公平锁 | 仅非公平 | 可选 |
| 条件变量 | 单个（wait/notify） | 多个（Condition） |
| 锁升级 | 有（偏向→轻量→重量） | 无 |
| 性能 | JDK 6 后优化很大 | 高竞争场景更优 |

## 锁优化思路总结

1. **减小锁粒度：** 只锁必要的代码段，如 ConcurrentHashMap 的分段锁
2. **锁分离：** 读写分离，如 ReadWriteLock
3. **锁消除：** JIT 编译器检测到不可能存在竞争时，消除不必要的锁
4. **锁粗化：** 如果一段代码中连续多次加锁解锁同一个对象，合并为一次
5. **无锁编程：** 使用 CAS、原子类等替代锁
