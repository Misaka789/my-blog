## I/O 设备分类

**按使用特性分：**

- **块设备：** 以数据块为单位传输，支持随机访问。如磁盘、SSD。
- **字符设备：** 以字符为单位传输，不支持随机访问。如键盘、鼠标、串口。

**按传输速率分：**

- 低速设备：键盘、鼠标
- 中速设备：打印机
- 高速设备：磁盘、网卡

## I/O 控制方式

### 1. 程序直接控制（轮询）

CPU 不断查询 I/O 设备状态，直到设备就绪后再传输数据。

```
while (设备未就绪) {
    // 忙等待
}
读/写数据
```

**缺点：** CPU 忙等待，利用率极低。

### 2. 中断驱动

CPU 发出 I/O 命令后继续执行其他任务。设备完成后发出**中断**通知 CPU。

**流程：**

1. CPU 向 I/O 设备发出命令
2. CPU 转去执行其他进程
3. 设备完成 I/O 操作
4. 设备发出中断请求
5. CPU 响应中断，执行中断处理程序
6. 传输数据（以字/字节为单位）

**优点：** CPU 不用忙等待

**缺点：** 每次只传输一个字/字节，频繁中断开销大

### 3. DMA（Direct Memory Access）

DMA 控制器直接在设备和内存之间传输数据，不经过 CPU。

**流程：**

1. CPU 向 DMA 控制器发出命令（起始地址、传输长度、方向）
2. DMA 控制器控制总线，在设备和内存间**批量传输数据**
3. 传输完成后，DMA 向 CPU 发出中断

**优点：** 以块为单位传输，CPU 只在开始和结束时参与

**DMA 控制器包含：**

- 内存地址寄存器（MAR）
- 数据计数器（DC）
- 数据缓冲寄存器
- 控制/状态寄存器

### 4. 通道控制

通道是一个专用的 I/O 处理器，可以执行通道程序来控制 I/O 操作。CPU 只需启动通道，通道自主完成复杂的 I/O 操作序列。

**I/O 控制方式对比：**

| 方式 | CPU 介入程度 | 传输单位 | 数据流向 |
|------|------------|----------|----------|
| 轮询 | 极高 | 字/字节 | 设备→CPU→内存 |
| 中断 | 中 | 字/字节 | 设备→CPU→内存 |
| DMA | 低 | 块 | 设备→内存 |
| 通道 | 极低 | 一组块 | 设备→内存 |

## 中断机制

### 中断的分类

**外中断（硬件中断）：**

- 来自 CPU 外部的事件
- I/O 中断：设备完成操作
- 时钟中断：时间片用完
- 硬件故障中断

**内中断（异常/陷入）：**

- CPU 内部执行指令时产生
- **陷入（Trap）：** 系统调用（如 `int 0x80`）
- **故障（Fault）：** 可恢复的错误，如缺页中断
- **终止（Abort）：** 不可恢复的错误，如除零

### 中断处理过程

1. CPU 检测到中断信号
2. 保存当前进程的上下文（PC、寄存器、PSW）
3. 根据中断向量表找到对应的中断处理程序
4. 切换到内核态
5. 执行中断处理程序
6. 恢复被中断进程的上下文（或调度新进程）

## 用户态与内核态

### 基本概念

- **用户态（User Mode）：** 只能执行非特权指令，不能直接访问硬件和内核数据
- **内核态（Kernel Mode）：** 可以执行所有指令，包括特权指令

### 为什么要区分？

保护操作系统不被用户程序破坏。用户程序不能直接操作硬件、修改页表、关中断等。

### 用户态 → 内核态的三种方式

1. **系统调用：** 用户程序主动请求操作系统服务（如 read、write、fork）
2. **异常：** 执行指令出错（如缺页、除零、非法地址）
3. **外部中断：** I/O 完成、时钟中断等

### 系统调用的过程

1. 用户程序调用库函数（如 `read()`）
2. 库函数将参数放入寄存器，执行 `syscall` / `int 0x80` 指令
3. CPU 切换到内核态
4. 根据系统调用号查找系统调用表
5. 执行对应的内核函数
6. 将结果返回用户态

## 五种 I/O 模型

以网络 I/O（socket read）为例，这是 Unix/Linux 下的五种经典 I/O 模型：

### 1. 阻塞 I/O（Blocking I/O）

```
用户进程  →  recvfrom()  →  阻塞等待  →  数据就绪  →  内核拷贝到用户空间  →  返回
```

- 最简单的模型
- 进程阻塞直到数据完全就绪并拷贝到用户空间
- 一个线程只能处理一个连接

### 2. 非阻塞 I/O（Non-blocking I/O）

```
用户进程  →  recvfrom()  →  立即返回 EAGAIN  →  反复轮询  →  数据就绪  →  拷贝  →  返回
```

- 设备未就绪时立即返回错误
- 用户进程需要不断轮询
- CPU 利用率低（忙等）

### 3. I/O 多路复用（I/O Multiplexing）

```
用户进程  →  select/poll/epoll()  →  阻塞等待多个 fd  →  某个 fd 就绪  →  recvfrom()  →  返回
```

- **一个线程可以监听多个 fd**
- `select`：有 1024 个 fd 上限，每次都要遍历所有 fd
- `poll`：无上限，但仍需遍历
- `epoll`：基于事件驱动，只返回就绪的 fd，性能最好

**select vs poll vs epoll：**

| 维度 | select | poll | epoll |
|------|--------|------|-------|
| fd 上限 | 1024 | 无限制 | 无限制 |
| 实现方式 | 遍历 | 遍历 | 回调/事件驱动 |
| fd 拷贝 | 每次全量拷贝 | 每次全量拷贝 | 只在注册时拷贝 |
| 时间复杂度 | O(n) | O(n) | O(1)（就绪事件） |
| 适用场景 | 连接少 | 连接少 | 高并发（如 Nginx） |

### 4. 信号驱动 I/O（Signal-driven I/O）

```
用户进程  →  注册 SIGIO 信号  →  继续执行  →  数据就绪时收到信号  →  recvfrom()  →  返回
```

- 内核在数据就绪时发送 SIGIO 信号
- 实际中较少使用

### 5. 异步 I/O（Asynchronous I/O）

```
用户进程  →  aio_read()  →  立即返回  →  继续执行  →  内核完成拷贝后通知用户进程  →  处理数据
```

- 用户进程发起请求后完全不阻塞
- 内核完成数据拷贝到用户空间后才通知
- **唯一真正的异步模型**
- Linux 的 `io_uring` 是现代高性能异步 I/O 接口

### 五种模型对比

| 模型 | 等待数据就绪 | 数据拷贝阶段 | 是否真异步 |
|------|------------|-------------|-----------|
| 阻塞 I/O | 阻塞 | 阻塞 | 否 |
| 非阻塞 I/O | 轮询 | 阻塞 | 否 |
| I/O 多路复用 | 阻塞在 select | 阻塞 | 否 |
| 信号驱动 | 非阻塞 | 阻塞 | 否 |
| 异步 I/O | 非阻塞 | 非阻塞 | 是 |

前四种都是**同步 I/O**，因为数据拷贝阶段进程都被阻塞。

## 零拷贝（Zero-Copy）

### 传统 I/O 的问题

从磁盘读文件发送到网络，传统方式需要 4 次拷贝 + 4 次上下文切换：

```
磁盘 → 内核缓冲区 → 用户缓冲区 → Socket 缓冲区 → 网卡
       (DMA拷贝)    (CPU拷贝)     (CPU拷贝)      (DMA拷贝)
```

### mmap + write

用 mmap 将内核缓冲区映射到用户空间，减少一次拷贝：

```
磁盘 → 内核缓冲区(映射到用户空间) → Socket 缓冲区 → 网卡
       (DMA拷贝)                    (CPU拷贝)      (DMA拷贝)
```

3 次拷贝，4 次上下文切换。

### sendfile

Linux 2.1 引入，直接在内核中从文件描述符传输到 Socket：

```
磁盘 → 内核缓冲区 → Socket 缓冲区 → 网卡
       (DMA拷贝)    (CPU拷贝)      (DMA拷贝)
```

3 次拷贝，2 次上下文切换。Nginx、Kafka 等高性能软件广泛使用。

### sendfile + DMA gather

如果网卡支持 scatter-gather DMA：

```
磁盘 → 内核缓冲区 ————————————→ 网卡
       (DMA拷贝)   (DMA gather)
```

2 次拷贝（都是 DMA），2 次上下文切换。真正的零 CPU 拷贝。

## 常见面试问题

**Q：什么是缓冲区？为什么需要缓冲？**

缓冲区是内存中的一块区域，用于暂存 I/O 数据。目的：
- 缓和 CPU 和 I/O 设备的速度差异
- 减少中断次数
- 解决数据粒度不匹配

**Q：select 和 epoll 的区别？**

select 每次调用都要传入所有 fd 集合，内核需要遍历所有 fd；epoll 通过 epoll_ctl 注册感兴趣的 fd，epoll_wait 只返回就绪的 fd，在高并发场景下性能远优于 select。

**Q：Reactor 和 Proactor 模式？**

- **Reactor：** 同步 I/O 多路复用 + 事件分发。用户线程负责读写。如 Netty、Redis。
- **Proactor：** 异步 I/O + 事件分发。内核完成读写后通知用户线程。如 Windows IOCP。
