设计 API 最容易陷入的一种错觉，是“字段命名正确、状态码合理、文档也写了，所以接口设计应该没有问题”。

但真正决定接口体验的，往往不是文档里那些显眼的部分，而是双方默认接受、却很少明确写出来的隐性契约。

## 什么是隐性契约

举几个很常见的例子：

- 某个字段虽然写着 optional，但前端默认一直会传。
- 某个状态虽然文档没限制，但后端只在两个场景下会返回。
- 某个列表接口看上去支持分页，其实排序规则并不稳定。

这些内容不一定会出现在文档里，但消费方一旦踩中，就会立刻感受到接口的“不顺手”。

## API 的问题通常暴露在协作阶段

很多设计在评审时看起来都成立，真正出问题往往是在联调的时候。

因为到了联调阶段，接口不再只是“定义”，而是开始和真实页面、真实业务流程、真实错误处理放在一起工作。这个时候，那些没写出来的默认前提就会陆续冒出来。

## 我现在会特别注意三类问题

### 命名是否稳定

字段名一旦发布出去，就会变成外部系统的依赖。命名摇摆不定，后面会带来持续的迁移成本。

### 边界是否清晰

接口应该尽量让人一眼看明白：哪些是必填，哪些是只读，哪些状态不会同时出现。

### 错误是否可处理

最糟糕的接口不是报错，而是报了一个消费方根本没法处理的错。一个好的错误响应应该能帮助调用方立刻进入修复路径。

## 好的 API 不是“功能齐全”，而是“预期稳定”

我越来越觉得，接口设计的核心不是把所有情况都覆盖到，而是让消费方形成稳定预期。

当一个接口的输入边界清晰、返回结构稳定、错误语义明确时，它就会自然变得更容易被信任，也更容易被长期维护。
