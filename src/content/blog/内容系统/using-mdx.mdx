Markdown 很适合长期写作，因为它足够轻，也足够稳定。但当内容逐渐变多以后，我会越来越明显地感受到：有些信息其实不只是“段落”，它们更像可以反复复用的内容模块。

## 我为什么会开始想用 MDX

最早吸引我的并不是组件，而是“内容可以拥有更清晰的结构”。

比如在一篇文章里，有些东西会反复出现：

- 提示卡片
- 对比信息
- 固定格式的实验记录
- 可复用的引用模块

如果每次都靠手写 Markdown 去模拟这些结构，时间久了就会很散。

## 真正有价值的不是炫技，而是可组合

我喜欢 MDX 的原因，不是因为它能把页面做得更复杂，而是因为它能让一部分内容表达变得更稳定。

### 什么时候值得上 MDX

对我来说，下面几种情况最适合：

- 某类信息会在很多文章里反复出现
- 内容本身有明确的结构模式
- 希望内容和页面组件之间保持轻量协作

如果只是普通长文，其实纯 Markdown 已经足够好。

## 内容系统的重点，还是长期维护

无论是 Markdown 还是 MDX，最后都绕不开一个问题：这个系统半年后还好不好维护。

所以我会更在意这些事：

1. 目录结构是不是清楚
2. 文章元数据是否统一
3. 分类是不是能自然长出来
4. 页面组件会不会反过来绑架写作

如果一套内容系统让写作本身变得更重，那它大概率就走偏了。

## 我想保留的，是写作的轻感

我希望未来这个博客里的文章既保留 Markdown 那种轻盈，也能在需要时拥有更完整的表达力。

所以对我来说，MDX 不是为了“做得更花”，而是为了在内容越来越多以后，依然能把结构整理得更清楚。
