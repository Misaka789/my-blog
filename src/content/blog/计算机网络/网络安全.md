## 网络安全的四大目标

1. **机密性（Confidentiality）：** 信息不被未授权者获取 → 加密
2. **完整性（Integrity）：** 信息在传输中不被篡改 → 哈希/MAC
3. **身份认证（Authentication）：** 确认通信方的身份 → 数字证书
4. **不可否认性（Non-repudiation）：** 发送方不能否认曾发送过信息 → 数字签名

## 加密体系

### 对称加密

加密和解密使用**同一个密钥**。

| 算法 | 密钥长度 | 特点 |
|------|---------|------|
| AES | 128/192/256 位 | 当前最主流，安全高效 |
| DES | 56 位 | 已不安全，被 AES 取代 |
| 3DES | 168 位 | DES 的增强版，仍在使用 |
| ChaCha20 | 256 位 | Google 推广，适合移动设备 |

**优点：** 速度快

**缺点：** 密钥分发困难（双方如何安全地共享密钥？）

### 非对称加密

使用**公钥和私钥**一对密钥。公钥加密只能私钥解密，反之亦然。

| 算法 | 特点 |
|------|------|
| RSA | 最广泛使用，基于大数分解 |
| ECC | 同等安全性下密钥更短，性能更好 |
| ECDHE | 基于 ECC 的密钥交换，支持前向安全 |

**优点：** 解决密钥分发问题

**缺点：** 速度慢（比对称加密慢 100-1000 倍）

### 混合加密

HTTPS 采用的方案：

1. 用**非对称加密**交换对称密钥（安全）
2. 用**对称加密**传输实际数据（高效）

## 哈希函数

将任意长度的输入映射为固定长度的输出（摘要），不可逆。

| 算法 | 输出长度 | 状态 |
|------|---------|------|
| MD5 | 128 位 | 已不安全（可碰撞） |
| SHA-1 | 160 位 | 已不安全 |
| SHA-256 | 256 位 | 当前主流 |
| SHA-3 | 可变 | 最新标准 |

**用途：**

- 密码存储（加盐哈希）
- 数据完整性校验
- 数字签名

## 数字签名

### 原理

用于验证消息的**完整性和发送方身份**。

```
发送方：
1. 对消息计算哈希值
2. 用自己的私钥加密哈希值 → 数字签名
3. 将消息 + 数字签名一起发送

接收方：
1. 用发送方的公钥解密签名，得到哈希值 A
2. 对收到的消息计算哈希值 B
3. 比较 A 和 B，相同则消息未被篡改且确实来自发送方
```

### 数字证书

**问题：** 怎么确认公钥确实属于声称的发送方？（中间人可能替换公钥）

**解决方案：** 数字证书 —— 由**证书颁发机构（CA）** 签发。

**证书内容：**

- 持有者信息
- 持有者公钥
- 颁发机构
- 有效期
- CA 的数字签名

**证书链：**

```
根 CA 证书（预装在浏览器/操作系统中）
  └── 中间 CA 证书
        └── 服务器证书
```

浏览器逐级验证签名，直到信任的根 CA。

## 常见 Web 攻击与防御

### XSS（跨站脚本攻击）

**原理：** 攻击者在网页中注入恶意脚本，当其他用户访问该页面时，脚本在其浏览器中执行。

**类型：**

- **反射型：** 恶意脚本在 URL 参数中，服务器直接返回给浏览器
- **存储型：** 恶意脚本存储在数据库中（如评论），所有访问者都会执行
- **DOM 型：** 前端 JavaScript 直接使用未经过滤的用户输入操作 DOM

**危害：** 窃取 Cookie/Token、劫持会话、钓鱼、篡改页面

**防御：**

1. **输出编码：** 对输出到 HTML 的内容进行转义（`<` → `<`）
2. **CSP（Content Security Policy）：** 限制页面可以加载的资源来源
3. **HttpOnly Cookie：** 禁止 JavaScript 访问 Cookie
4. **输入验证：** 对用户输入进行白名单过滤

### CSRF（跨站请求伪造）

**原理：** 攻击者诱导已登录用户访问恶意页面，该页面自动向目标网站发起请求（浏览器会自动带上 Cookie）。

```
用户已登录 bank.com
攻击者页面包含：<img src="https://bank.com/transfer?to=attacker&amount=10000">
浏览器自动携带 bank.com 的 Cookie 发起请求
```

**防御：**

1. **CSRF Token：** 服务器生成随机 Token，表单提交时验证
2. **SameSite Cookie：** 设置 `SameSite=Strict` 或 `Lax`，限制跨站发送 Cookie
3. **检查 Referer/Origin 头：** 验证请求来源
4. **关键操作二次确认：** 如短信验证码

### SQL 注入

**原理：** 攻击者在输入中嵌入 SQL 代码，服务器未经过滤直接拼接到 SQL 语句中执行。

```sql
-- 用户输入用户名: ' OR '1'='1
-- 拼接后的 SQL:
SELECT * FROM users WHERE username = '' OR '1'='1' AND password = ''
-- 条件永远为真，绕过认证
```

**防御：**

1. **参数化查询（预编译语句）：** 将用户输入作为参数而非 SQL 代码
2. **ORM 框架：** 自动处理参数化
3. **输入验证：** 限制输入的字符和长度
4. **最小权限原则：** 数据库用户只给必要的权限

### 中间人攻击（MITM）

**原理：** 攻击者插入通信双方之间，分别与双方建立独立连接，转发并可能篡改消息。

```
客户端 ←→ 攻击者 ←→ 服务器
```

**防御：**

- HTTPS（TLS 加密 + 证书验证）
- 证书锁定（Certificate Pinning）
- HSTS（强制使用 HTTPS）

## DDoS 攻击

### 什么是 DDoS

**分布式拒绝服务攻击（Distributed Denial of Service）：** 利用大量被控主机（僵尸网络）同时向目标发送海量请求，耗尽服务器资源或带宽。

### 攻击类型

**网络层攻击：**

- **SYN Flood：** 发送大量 SYN 请求不完成握手，耗尽服务器半连接队列
- **UDP Flood：** 发送大量 UDP 包，耗尽带宽
- **ICMP Flood（Ping Flood）：** 大量 ping 请求

**应用层攻击：**

- **HTTP Flood：** 模拟正常 HTTP 请求，难以区分
- **CC 攻击：** 针对消耗资源大的页面（如搜索、数据库查询）

### 防御

- **流量清洗：** CDN/DDoS 防护服务过滤恶意流量
- **SYN Cookie：** 不为半连接分配资源
- **限流：** 对单 IP 的请求速率进行限制
- **黑洞路由：** 极端情况下丢弃所有流往目标的流量
- **弹性扩容：** 云服务自动扩容应对流量高峰

## 其他安全机制

### 防火墙

- **包过滤防火墙：** 根据 IP、端口、协议过滤（网络层/传输层）
- **状态检测防火墙：** 跟踪连接状态，只允许属于合法连接的包通过
- **应用层防火墙（WAF）：** 检查 HTTP 内容，防御 XSS、SQL 注入等

### VPN

**虚拟专用网络（Virtual Private Network）：** 通过加密隧道在公共网络上建立安全的专用连接。

常见协议：IPSec、OpenVPN、WireGuard

### OAuth 2.0

- 授权框架，允许第三方应用在用户授权下访问其资源
- 用户不需要将密码给第三方
- 通过 Access Token 控制权限范围和有效期

## 常见面试问题

**Q：HTTPS 能防止中间人攻击吗？**

能，前提是正确验证了证书。如果用户忽略了证书警告或系统被安装了恶意根证书，HTTPS 也可能被中间人攻击。

**Q：Cookie 被 XSS 窃取后怎么办？**

设置 HttpOnly 防止 JS 读取 Cookie；使用 CSP 防止脚本注入；Token 可以存在 localStorage 中但需防范 XSS；关键操作增加二次验证。

**Q：对称加密和非对称加密在 HTTPS 中分别用在哪里？**

非对称加密用于 TLS 握手阶段的密钥交换和身份认证，对称加密用于握手后的实际数据传输。

**Q：什么是前向安全（Forward Secrecy）？**

即使服务器的长期私钥泄露，过去的通信记录也无法被解密。ECDHE 密钥交换算法支持前向安全，因为每次会话使用临时密钥对，会话结束后密钥销毁。
