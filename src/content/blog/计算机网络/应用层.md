## HTTP 协议

### 基本概念

HTTP（HyperText Transfer Protocol）是基于 TCP 的应用层协议，采用**请求-响应**模式。

**特点：**

- 无状态：每次请求独立，服务器不保留之前请求的信息
- 无连接（HTTP/1.0）：每次请求完成后断开连接
- 简单灵活：请求方法、URI、协议版本都是文本

### HTTP 报文结构

**请求报文：**

```
GET /index.html HTTP/1.1          ← 请求行（方法 + URI + 版本）
Host: www.example.com              ← 请求头
Connection: keep-alive
Accept: text/html
                                   ← 空行
请求体（GET 一般没有）              ← 请求体
```

**响应报文：**

```
HTTP/1.1 200 OK                    ← 状态行（版本 + 状态码 + 原因短语）
Content-Type: text/html            ← 响应头
Content-Length: 1234
                                   ← 空行
<html>...</html>                   ← 响应体
```

### HTTP 请求方法

| 方法 | 语义 | 幂等 | 安全 | 请求体 |
|------|------|------|------|--------|
| GET | 获取资源 | 是 | 是 | 无 |
| POST | 提交数据/创建资源 | 否 | 否 | 有 |
| PUT | 替换/更新整个资源 | 是 | 否 | 有 |
| DELETE | 删除资源 | 是 | 否 | 无 |
| PATCH | 部分更新资源 | 否 | 否 | 有 |
| HEAD | 获取响应头（不要响应体） | 是 | 是 | 无 |
| OPTIONS | 查询服务器支持的方法 | 是 | 是 | 无 |

**幂等：** 多次执行同一请求，结果相同。

**安全：** 不会修改服务器状态。

### GET vs POST

| 维度 | GET | POST |
|------|-----|------|
| 参数位置 | URL 查询字符串 | 请求体 |
| 长度限制 | 受浏览器/服务器 URL 长度限制 | 无限制（理论上） |
| 缓存 | 可以被缓存 | 一般不缓存 |
| 幂等性 | 幂等 | 非幂等 |
| 安全性 | 参数暴露在 URL | 相对安全（仍是明文） |
| 浏览器回退 | 无害 | 会重新提交 |

### HTTP 状态码

| 分类 | 范围 | 含义 | 常见状态码 |
|------|------|------|-----------|
| 1xx | 100-199 | 信息性 | 100 Continue |
| 2xx | 200-299 | 成功 | 200 OK, 201 Created, 204 No Content |
| 3xx | 300-399 | 重定向 | 301 永久重定向, 302 临时重定向, 304 Not Modified |
| 4xx | 400-499 | 客户端错误 | 400 Bad Request, 401 未授权, 403 禁止, 404 Not Found |
| 5xx | 500-599 | 服务器错误 | 500 Internal Error, 502 Bad Gateway, 503 Service Unavailable |

**301 vs 302：**

- 301 永久重定向：浏览器会缓存，下次直接访问新地址
- 302 临时重定向：浏览器不缓存，每次仍请求原地址

**304 Not Modified：** 协商缓存命中，服务器告诉客户端可以使用本地缓存。

### HTTP 缓存机制

**强缓存：** 不需要请求服务器，直接使用本地缓存。

- `Cache-Control: max-age=3600`（HTTP/1.1，优先级高）
- `Expires: Thu, 01 Dec 2026 16:00:00 GMT`（HTTP/1.0）

**协商缓存：** 需要请求服务器验证缓存是否有效。

- `Last-Modified / If-Modified-Since`：基于文件修改时间
- `ETag / If-None-Match`：基于文件内容哈希（优先级高）

```
第一次请求：服务器返回 ETag: "abc123"
后续请求：客户端发送 If-None-Match: "abc123"
未变化：服务器返回 304
已变化：服务器返回 200 + 新内容
```

## HTTP 版本演进

### HTTP/1.0

- 每次请求都要新建 TCP 连接（短连接）
- 无 Host 头，一个 IP 只能一个网站

### HTTP/1.1

- **持久连接（Keep-Alive）：** 默认开启，复用 TCP 连接
- **管道化（Pipelining）：** 可以连续发送多个请求，不用等上一个响应
- 增加 Host 头，支持虚拟主机
- 增加 `Transfer-Encoding: chunked` 分块传输

**队头阻塞（Head-of-Line Blocking）：** 虽然支持管道化，但响应必须按请求顺序返回。前一个响应慢会阻塞后面所有响应。

### HTTP/2

- **多路复用（Multiplexing）：** 一个 TCP 连接上可以并行传输多个请求和响应，彻底解决 HTTP 层的队头阻塞
- **二进制分帧：** 将数据分成更小的帧（Frame），帧可以乱序发送
- **头部压缩（HPACK）：** 压缩请求/响应头，减少传输量
- **服务器推送（Server Push）：** 服务器可以主动推送资源

**仍然存在的问题：** TCP 层的队头阻塞（一个 TCP 包丢失会阻塞所有流）

### HTTP/3

- 基于 **QUIC 协议**（UDP 之上）
- 解决了 TCP 层的队头阻塞
- 内建 TLS 1.3 加密
- 连接迁移：网络切换时不需要重新握手（基于 Connection ID）
- 0-RTT 建连

## HTTPS

### 为什么需要 HTTPS

HTTP 是明文传输，存在三大风险：

- **窃听：** 第三方可以截获通信内容
- **篡改：** 第三方可以修改通信内容
- **冒充：** 第三方可以伪装成通信方

### HTTPS = HTTP + TLS/SSL

在 HTTP 和 TCP 之间加入 TLS（Transport Layer Security）层：

```
HTTP → TLS → TCP → IP
```

**HTTPS 提供：**

- **加密：** 防窃听（对称加密传输数据）
- **完整性：** 防篡改（消息认证码 MAC）
- **身份认证：** 防冒充（数字证书）

### TLS 握手过程（TLS 1.2，RSA）

```
客户端                          服务器
  |--- ClientHello ------------->|  （支持的 TLS 版本、加密套件、随机数 Client Random）
  |<-- ServerHello --------------|  （选定的加密套件、随机数 Server Random）
  |<-- Certificate --------------|  （服务器证书，含公钥）
  |<-- ServerHelloDone ----------|
  |--- ClientKeyExchange ------->|  （用服务器公钥加密 Pre-Master Secret）
  |--- ChangeCipherSpec -------->|  （后续使用协商的密钥加密）
  |--- Finished ---------------->|
  |<-- ChangeCipherSpec ---------|
  |<-- Finished -----------------|
  |<======= 加密通信 ==========>|
```

**密钥生成：** Master Secret = f(Client Random, Server Random, Pre-Master Secret)，然后派生出对称加密密钥。

### 对称加密 vs 非对称加密

| 维度 | 对称加密 | 非对称加密 |
|------|---------|-----------|
| 密钥 | 加密解密用同一个密钥 | 公钥加密，私钥解密 |
| 速度 | 快 | 慢（约慢 100-1000 倍） |
| 典型算法 | AES、ChaCha20 | RSA、ECDHE |
| HTTPS 中的用途 | 加密实际数据传输 | 密钥交换/身份认证 |

**HTTPS 的混合加密：** 用非对称加密交换对称密钥，再用对称密钥加密后续通信。

## DNS

### DNS 是什么

DNS（Domain Name System）负责将域名解析为 IP 地址。

### 域名层级

```
www.example.com.
 |      |     |  |
三级  二级  顶级 根
```

### DNS 解析过程

1. 浏览器缓存 → 操作系统缓存 → hosts 文件
2. 本地 DNS 服务器（递归查询）
3. 本地 DNS → 根 DNS 服务器（返回顶级域 DNS 地址）
4. 本地 DNS → 顶级域 DNS 服务器（返回权威 DNS 地址）
5. 本地 DNS → 权威 DNS 服务器（返回最终 IP）
6. 本地 DNS 缓存结果，返回给客户端

**递归查询 vs 迭代查询：**

- 客户端向本地 DNS 发起**递归查询**（帮我查到底）
- 本地 DNS 向其他 DNS 服务器发起**迭代查询**（告诉我下一步问谁）

### DNS 记录类型

| 类型 | 含义 | 示例 |
|------|------|------|
| A | 域名 → IPv4 地址 | example.com → 1.2.3.4 |
| AAAA | 域名 → IPv6 地址 | example.com → 2001:db8::1 |
| CNAME | 域名别名 | www.example.com → example.com |
| MX | 邮件服务器 | example.com → mail.example.com |
| NS | 域名服务器 | example.com → ns1.example.com |
| TXT | 文本记录 | SPF、DKIM 验证 |

## Cookie 与 Session

### Cookie

- 存储在**客户端浏览器**中的小文本
- 每次请求自动携带在 `Cookie` 头中
- 有大小限制（一般 4KB）
- 可设置过期时间、域、路径、安全标志

```
Set-Cookie: sessionId=abc123; Path=/; HttpOnly; Secure; SameSite=Strict
```

**重要属性：**

- `HttpOnly`：禁止 JavaScript 访问，防 XSS 窃取
- `Secure`：仅通过 HTTPS 传输
- `SameSite`：限制跨站发送，防 CSRF

### Session

- 存储在**服务器端**
- 通过 Session ID（通常存在 Cookie 中）标识用户
- 安全性更高（数据不暴露给客户端）
- 占用服务器资源

### Cookie vs Session

| 维度 | Cookie | Session |
|------|--------|---------|
| 存储位置 | 客户端 | 服务器 |
| 安全性 | 较低 | 较高 |
| 大小限制 | ~4KB | 无限制 |
| 生命周期 | 可持久化 | 默认会话结束失效 |
| 服务器压力 | 无 | 有（需要存储） |

### Token（JWT）

- 无状态的认证方案，服务器不需要存储会话信息
- Token 包含用户信息和签名，服务器通过验证签名确认身份
- 适合分布式系统（不需要 Session 共享）

## WebSocket

- 全双工通信协议，建立在 TCP 之上
- 通过 HTTP 升级握手建立连接
- 连接建立后，客户端和服务器可以随时互发消息
- 适合实时场景：聊天、推送、协作编辑

```
客户端：GET / HTTP/1.1
        Upgrade: websocket
        Connection: Upgrade

服务器：HTTP/1.1 101 Switching Protocols
        Upgrade: websocket
```

## 常见面试问题

**Q：HTTP 是无状态的，怎么保持登录状态？**

通过 Cookie + Session 或 Token（JWT）。Cookie 携带 Session ID 或 Token，服务器据此识别用户身份。

**Q：HTTP/2 的多路复用和 HTTP/1.1 的 Keep-Alive 有什么区别？**

Keep-Alive 只是复用 TCP 连接，请求仍需串行处理（有队头阻塞）。HTTP/2 的多路复用允许在同一个连接上并行交错发送多个请求和响应。

**Q：HTTPS 比 HTTP 慢多少？**

主要多了 TLS 握手的开销（1-2 个 RTT）。TLS 1.3 支持 1-RTT 握手，甚至 0-RTT 恢复。数据传输阶段因为对称加密，性能损耗很小。

**Q：DNS 使用 TCP 还是 UDP？**

通常使用 UDP（端口 53），因为查询报文小、速度快。当响应超过 512 字节（如区域传送）时使用 TCP。
