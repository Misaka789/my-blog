当我第一次接触 MDX 时，我的第一反应是"这不就是 Markdown 里能写 JSX 吗？"但用了一年多之后，我发现 MDX 真正的价值不在于"能写 JSX"，而在于它打破了**内容和交互之间的人为边界**。

## MDX 解决的真实问题

传统技术博客有一个尴尬的困境：你想要展示一段代码的运行效果，但 Markdown 只能给你静态的代码块。于是你要么贴一张截图（信息损失），要么让读者"自己跑一下试试"（摩擦成本）。

MDX 让你可以把一个活的组件直接嵌入内容流中：

import HeaderLink from '../../components/HeaderLink.astro';

<HeaderLink href="#" onclick="alert('这就是 MDX 的魅力——内容即交互')">
	点击体验：嵌入在文章中的交互组件
</HeaderLink>

这不只是"炫技"。对于技术写作来说，**可交互的示例价值远高于静态描述**。一个可以点击、输入、看到即时反馈的 demo，胜过一千字的文字说明。

## Astro + MDX 的工程优势

在 Astro 中使用 MDX 有一个独特的优势：**默认零 JS 运行时**。

```astro
---
// 这个组件在构建时执行，不会产生客户端 JS
import DataTable from '../components/DataTable.astro';
---

<DataTable data={fetchedData} />
```

大多数博客中的"交互组件"其实不需要客户端交互性——一个格式化的数据表格、一个语法高亮的代码对比、一个带锚点的目录导航。这些用 Astro 组件实现，构建时渲染，零运行时开销。

只有真正需要客户端交互的组件（比如可编辑的代码沙箱、动态图表），才需要加上 `client:` 指令：

```astro
<!-- 只有这个组件会附带 JS -->
<InteractiveChart client:visible data={chartData} />
```

这种"选择性水化"的策略，让你的博客既有丰富的交互能力，又保持了极致的加载性能。

## 内容架构的思考

MDX 带来的另一个深层价值是**内容的可组合性**。当你的博客文章可以引用组件，那么重复出现的内容模式就可以被抽象：

```mdx
import Callout from '../components/Callout.astro';
import TechStack from '../components/TechStack.astro';

<Callout type="warning">
  这个 API 在 v3.0 中将被废弃，请迁移到新的实现。
</Callout>

<TechStack items={['TypeScript', 'Astro', 'MDX']} />
```

这意味着你的博客不再是一堆孤立的 Markdown 文件，而是一个**有统一设计语言的内容系统**。更新一个 Callout 组件的样式，所有文章中的提示框都会同步更新。

## 实践建议

经过一年多的 MDX 博客实践，几条经验：

1. **不要为了用 MDX 而用 MDX。** 如果一篇文章纯文字就能表达清楚，普通 Markdown 就够了。MDX 的价值在于"需要的时候能用"，而不是"每篇都得用"。

2. **组件 API 要简单。** 博客组件不是 UI 库组件。一个接受 3 个以上 props 的博客组件就太复杂了——写文章的时候不想查组件文档。

3. **注意构建性能。** MDX 的编译比普通 Markdown 慢。如果文章数量超过 100 篇，考虑用增量构建策略。

4. **保持内容的可移植性。** 不要让文章过度依赖自定义组件。好的 MDX 文章在去掉所有组件后，纯文本仍然可读。
